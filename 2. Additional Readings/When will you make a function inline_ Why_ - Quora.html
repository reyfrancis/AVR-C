Originally Answered: When will you make a function inline, and why?

inline is a suggestion to the compiler only - and modern compilers will ignore it if the function cant be inlined.

In my experience, it is best used for very short function which are called very often; if the compiler does decide to inline a function it will basically insert the code for the function directly into the calling code - thus eliminating the cpu time taken to execute the function calling overhead.

Bear in-mind that the calling overhead is pretty low; a new stack frame is created, the return address, registers and arguments are pushed to the stack. For a simple function this can be very low - a few instructions at most.

There is very little benefit from inlining a function where the execution time of the function is significantly longer than the calling overhead, or where the function isnt called that often; and modern compilers are able to detect these cases.

Originally Answered: When do we use an inline function, and why?

In C there may be an advantage to inlining some code lines, but generally not in C++.
The C++ compiler can determine if inlining is appropriate and do so automatically. Most of the time specifying inline in C++ code is ignored by the C++ compiler and it does it anyway if it will help or doesnt if it is detrimental.

C inlining means to put the body of the function in line with the place it is called instead of making a function call. The idea is that it will be faster to just duplicate the code and run it in sequence than to form a frame and a call stack.
So calling a function inside another will put some code for calling the function into your function, inlining it will compile the actual commands in your function to be executed.

Unless you have a good handle on how the C compiler actually generates assembler and how the CPU caches instructions and registers, you are probably going to lose efficiency on all but the tiniest functions.
Inline functions cannot really be inlined if they call any other functions, even the libc ones. The libc functions are already inlined where appropriate.

You can always do metrics on your code by testing both ways, 10% will be faster, 60% will be the same, 40% will be slower. You decide.

If you have a function that merely does some simple calculations or runs a tight for loop and returns, then it may be a candidate for inlining, but if it calls other functions or is recursive, or something, no.

Trust the compiler to do the right thing. Then test to see if it did.
